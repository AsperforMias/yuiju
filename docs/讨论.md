# 讨论

## 记忆模块（graphiti_core）接入讨论总结

### 背景与现状

- 聊天侧（Node）：当前对话历史仅保存在内存中（小时级窗口 + 条数限制），未做长期可检索记忆沉淀。
- 世界侧（Node）：虚拟世界的行为流水会写入 MongoDB（生产环境），用于回放/上下文，但目前主要是“最近 N 条”读取，不具备跨天的高质量检索与归纳能力。
- Python：仓库内已有 `packages/memory`（graphiti_core demo 代码），具备引入 Graphiti/Neo4j 的可行性。

### 1) 虚拟世界行为记录是否需要交给 graphiti_core 处理？

结论：需要处理，但不建议把“所有 tick 级流水”原封不动入图谱，建议分层。

- MongoDB：继续保存全量行为流水（审计、回放、统计、调试）。
- Graphiti（图谱记忆）：只接收“高价值 episode”，用于跨天、跨场景的语义检索与关系推理。
- 高价值 episode 示例：
  - 计划/目标变化（长期计划、短期计划的更新与原因）
  - 重要事件与结果（购物、学习、社交、获得/消耗物品等）
  - 与用户关系/偏好相关的稳定事实
- 不建议入图的噪声示例：Idle/重复的低信息量行为、无稳定事实的瞬态日志。

### 2) 记忆写入策略：实时处理还是分批处理？

推荐策略：同步落库原始事件 + 异步批处理入图（为主）+ 关键事实准实时入图（为辅）。

- 同步阶段（低延迟优先）：
  - 聊天：保存 user/assistant 的对话回合到数据库（原始事件）。
  - 世界：保存行为记录到数据库（原始事件）。
- 异步阶段（质量与成本优先）：
  - 由后台 worker/任务定期拉取“待处理事件”，聚合成语义自洽的 episode 后再写入 Graphiti。
  - 批处理粒度建议：按“对话回合”或“行为完成事件”聚合，而不是按单条 message/tick 直接入图。
- 准实时补充（召回新鲜度优先，可选）：
  - 对明确偏好/设定、身份锚点、关系变化等“高价值事实”可立即入图，确保刚说完就可被检索到。

#### token 开销与实时性评估（基于当前频率）

前提：Graphiti 写入 episode 往往会触发多次 LLM/Embedding 处理，因此成本应按“每个 episode 的总处理 token”估算，而不是只看原始文本长度。

- 世界侧频率：平均 30 分钟完成一次行为，约 48 条/天
  - 若每次行为完成就写入 1 条 episode，整体 token 开销通常在“可控”范围内
- 聊天侧频率：每天约 50 条消息（即时通讯，无固定对话轮）
  - 不建议“每条消息都写入 1 条 episode”（固定开销会被放大）
  - 建议将多条消息合并成更少的“会话片段 episode”，显著降低 token 与写入次数

#### 聊天侧策略（方案 1：按时间窗合并，准实时写入）

结论：聊天侧采用“按时间窗合并”的准实时写入策略，兼顾实现简单与 token 成本。

- 落库：每条消息先作为原始事件落库（用于审计、回放、失败重试）
- 合并：将一段时间内连续的多条消息合并为 1 条 Graphiti episode（会话片段）
  - 建议默认时间窗：10 分钟（可配置）
  - 当超过时间窗无新消息，则“结算”上一个片段并写入 Graphiti
- 关键事实加速（可选）：
  - 当检测到偏好/设定/关系变化等高价值内容时，可立即结算并写入，提升记忆新鲜度

### 3) NodeJS 与 Python graphiti_core 的集成方式选择

结论：选择方案 A（推荐）——独立 HTTP Memory Service。

#### 方案 A：独立 HTTP Memory Service（选定）

- 形态：Python 常驻服务进程，对外提供 HTTP API，封装 graphiti_core 的写入与检索能力。
- 能力边界：
  - 写入：add_episode（支持文本/结构化 JSON 的 episode）
  - 检索：search（支持按 user 维度隔离、可选 center node rerank 等能力）
- Node 调用方式：
  - 聊天侧：作为工具检索接口（例如替换/扩展现有的 memorySearch 工具），在生成回复前拉取相关记忆。
  - 写入侧：对话结束/行为完成后异步推送“待处理事件”或由 worker 批量推送 episode。
- 优势：
  - 复用 Neo4j/Graphiti 连接，性能与稳定性更可控
  - Node 无需处理 Python 运行时细节，边界清晰，便于扩展鉴权/限流/监控
  - 既能支持写入解耦（批处理），也能支持低延迟检索（聊天实时）

#### 方案 B：跨语言调用（未选）

- 形态：Node 通过子进程/脚本调用 Python。
- 风险：进程启动成本高、连接不可复用、并发与超时控制复杂，不适合聊天实时检索链路。

### 补充关键决策点（后续落地需要对齐）

- 记忆优先服务对象：聊天回复（message）与世界决策（world）是否都要接入。
- 记忆注入方式：走“工具检索”还是写入 system prompt（或组合）。
- 用户隔离主键：建议优先使用稳定标识（如 user_id），昵称可作为展示字段或辅助索引。

### 安全与配置注意事项

- graphiti/Neo4j/LLM/Embedding 的密钥与连接信息应使用环境变量注入，避免写死在代码中。
