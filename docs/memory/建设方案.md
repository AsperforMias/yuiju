# 记忆系统建设方案（最优解）

本方案以“长期可演化认知系统”为目标，而非仅构建可检索的记忆库。

## 1. 建设目标

- 支撑“悠酱”在回答与行动决策中的长期一致性与可解释性。
- 区分“历史真相”“当前信念”“稳定语义关系”，避免语义污染。
- 允许冲突事实并存，系统基于证据与时序动态选择当前信念。
- 保证每条高层结论都可追溯到原始事件证据。
- 通过治理机制控制图规模、噪声与长期退化。

## 2. 总体架构（四层模型）

### 2.1 Event Sourcing（真相层）

- 定义：不可变事件流（append-only），是唯一历史真相源。
- 存储：MongoDB（事件集合）+ Redis（短期窗口缓存）。
- 事件来源：行动执行结果、关键对话结论、关系/情绪变化、外部输入。
- 核心原则：事件可追加、不可覆盖、不可物理删除（仅归档标记）。

### 2.2 Temporal Knowledge Graph（语义层）

- 定义：提纯后的语义事实图，用于关系推断与长期知识表达。
- 存储：Neo4j。
- 抽取：Graphiti 负责候选实体/关系抽取。
- 核心原则：
  - 图只接收“事实候选”，不直接接收原始事件。
  - 每条事实必须绑定证据事件 ID 列表。
  - 采用双时间建模：
    - `valid_time`：事实在世界中成立的时间区间。
    - `system_time`：系统写入/更新该事实的时间区间。

### 2.3 Belief State（信念层）

- 定义：在存在冲突、噪声、时效变化时，系统对“当前成立事实”的计算结果。
- 形态：可物化为独立集合/表，或由规则引擎实时计算并缓存。
- 职责：
  - 置信度计算。
  - 冲突并存与裁决。
  - 反向记忆（否定证据）降权。
  - 输出“当前信念快照”供决策和回答。

### 2.4 Retrieval Orchestrator（检索决策层）

- 定义：统一检索编排层，负责问题分类、路由、融合和可解释输出。
- 核心能力：
  - 问题类型识别（事件回溯/长期事实/关系推断/混合问题）。
  - 多源检索（事件层、图层、信念层、短期上下文）。
  - 结果融合排序（相关性 + 时序 + 证据质量 + 信念置信度）。
  - 输出证据链和置信度。

## 3. 数据流设计

### 3.1 写入链路

1. 事件进入 Event Sourcing。
2. `Memory Event Ingestor` 做结构化与基础去噪。
3. `Fact Distiller` 从事件中提取事实候选。
4. `Candidate Validator` 做质量过滤（阈值、模式校验、重复检测）。
5. 候选事实进入 Graphiti 抽取增强后写入 Neo4j。
6. `Conflict Resolver` 更新 Belief State。
7. 生成可查询的信念快照缓存。

### 3.2 检索链路

1. `Query Classifier` 识别任务类型。
2. `Retrieval Orchestrator` 执行路由：
   - 事件问题：优先事件层。
   - 偏好/关系/长期事实：优先图层 + 信念层。
   - 混合问题：事件层定位时段 + 图层补全语义。
3. `Answer Composer` 输出答案、证据引用、置信度、时间锚点。

## 4. 核心数据模型

### 4.1 Event（事件真相）

建议字段：

- `event_id`：全局唯一 ID。
- `tick_id`：引擎 tick 标识。
- `actor`：行为主体（通常为悠酱）。
- `participants`：参与者列表。
- `event_type`：行为/对话/关系变化/情绪变化/外部事件。
- `time_at`：事件发生时间。
- `location`：场景位置。
- `intent`：行为意图。
- `result`：执行结果。
- `metadata`：扩展参数。
- `importance`：重要性分级。
- `source`：数据来源（action、dialog、system、external）。
- `raw_text`：原始描述文本。

### 4.2 Fact Candidate（事实候选）

建议字段：

- `candidate_id`
- `fact_type`：`identity | preference | relation | plan | habit | constraint`
- `subject/predicate/object`
- `qualifiers`：条件、频率、上下文限定。
- `confidence_initial`
- `evidence_event_ids`
- `valid_from/valid_to`
- `ttl_policy`

### 4.3 Graph Fact（图事实）

建议字段：

- `fact_id`
- `triple`
- `valid_time_range`
- `system_time_range`
- `version`
- `is_current`
- `evidence_event_ids`
- `confidence_graph`
- `conflict_group_id`

### 4.4 Belief（当前信念）

建议字段：

- `belief_id`
- `canonical_fact_key`
- `current_value`
- `confidence_final`
- `supporting_fact_ids`
- `opposing_fact_ids`
- `last_recomputed_at`
- `decision_trace`

## 5. 冲突处理与信念计算

### 5.1 冲突识别

- 同一 `canonical_fact_key` 下语义互斥的事实进入同一冲突组。
- 冲突事实不物理删除，保留并标注状态。

### 5.2 置信度计算（建议）

综合评分可由以下信号加权：

- 近期性（recency）
- 证据质量（来源可靠度、事件重要度）
- 一致性（是否与历史稳定模式一致）
- 多证据支持度（独立证据数量）
- 否定证据惩罚（反向记忆）

### 5.3 裁决策略

- 输出“当前成立值” + “备选值”。
- 当最高分低于阈值时，输出“不确定”，避免幻觉式确定回答。
- 对行为决策场景，支持 conservative 模式（优先高置信、低风险结论）。

## 6. Graphiti 与 Neo4j 的职责边界

- Graphiti：候选抽取与关系解析增强，不直接承担最终真值裁定。
- Neo4j：存储语义图与时态关系，支撑关系查询与路径推理。
- 业务侧模块：必须实现 `Fact Distiller`、`Conflict Resolver`、`Versioning/Governance`。

结论：Neo4j + Graphiti 是图语义层的合适组合，但必须由业务治理层补齐“可控性”。

## 7. 检索与回答策略

### 7.1 问题路由

- “什么时候做了什么”：事件层。
- “喜欢什么/与谁关系如何”：信念层 + 图层。
- “最近为什么这样做”：事件层（近期链路）+ 信念层（长期偏好）。

### 7.2 结果融合

排序分数建议综合：

- 语义相关性
- 时间贴近度
- 证据完整性
- 信念置信度
- 与当前状态一致性

### 7.3 可解释输出

每次输出附带：

- `answer`
- `confidence`
- `evidence_refs`（事件 ID / 图事实 ID）
- `time_anchor`
- `reasoning_summary`（简洁规则链）

## 8. 遗忘与治理

### 8.1 遗忘策略

- 真相层：不删，仅归档。
- 图层：低命中低价值节点降权、压缩、归档。
- 信念层：过期信念自动失活，等待新证据激活。

### 8.2 治理任务（定时）

- 图规模巡检：节点/边增长率、孤立节点比例。
- 命中率巡检：检索命中、引用率、回答使用率。
- 冲突巡检：高冲突实体列表与人工抽检队列。
- 质量巡检：高置信错误样本回放。

## 9. 质量评测体系

### 9.1 离线评测

- 事件回溯准确率（Event QA）。
- 偏好一致性（Preference Consistency）。
- 关系推断准确率（Relation Inference）。
- 冲突裁决正确率（Conflict Resolution Accuracy）。
- 可解释性完整度（Evidence Coverage）。

### 9.2 在线评测

- 决策稳定性（跨天行为一致度）。
- 回答漂移率（同类问题答案波动）。
- 检索延迟与成本。
- 用户纠错后收敛速度。

## 10. 工程模块划分

- `memory-event-ingestor`：事件标准化、去噪、写入真相层。
- `memory-fact-distiller`：事实候选抽取与质量过滤。
- `memory-graph-writer`：Graphiti + Neo4j 写入与版本管理。
- `memory-conflict-resolver`：冲突组管理与信念计算。
- `memory-retrieval-orchestrator`：路由、融合、解释输出。
- `memory-governance`：巡检、降权、归档、质量报告。

## 11. 分阶段实施里程碑（推荐）

### 阶段 A：真相层与候选层

- 完成事件模型统一与 append-only 写入。
- 完成 Fact Distiller 与候选质量阈值。
- 建立可追溯证据链。

### 阶段 B：图层与信念层

- 完成 Neo4j 图事实时态版本化。
- 完成冲突组与信念计算。
- 接入问答/决策流程。

### 阶段 C：检索编排与治理

- 完成路由器与融合排序。
- 完成治理任务与指标看板。
- 建立离线回放与持续评测。

## 12. 关键设计原则（必须遵守）

- 原始事件不直接入图。
- 高层结论必须可追溯到证据。
- 冲突事实允许并存，禁止简单覆盖抹历史。
- 真相层不可变，信念层可演化。
- 检索输出必须携带置信度与时间锚点。
