# 记忆模块规划

规划记忆模块应该如何建设。

## 建设目标

- 支持“悠酱”回答与行动决策的记忆支撑
- 避免记录噪声，保证可检索性与可解释性
- 覆盖事件回溯、偏好稳定、关系推断、状态变化

## 分层结构

- 短期记忆（Working）：最近对话 + 最近一次 tick 事件，供即时上下文；存 Redis/内存即可
- 事件记忆（Episodic）：可追溯事件片段，回答“什么时候做了什么”；仅存 MongoDB/现有记忆服务
- 事实记忆（Semantic/Graph）：稳定偏好、关系、长期事实；存 Graphiti/Neo4j
- 经验记忆（Patterns，可选）：行为统计与习惯模型，影响行动选择；按天/周聚合存 MongoDB

## 写入策略

- 只写关键事件：行为完成、重要对话结论、情绪/关系变化
- 结构化写入：who/when/where/what/why/result + type + importance
- 去重与合并：同天同类事件合并为 summary episode
- 记忆输入整理：从事件记忆中筛选关键文本与去噪，送入 Graphiti
- Graphiti 抽取：Graphiti 接收文本并自动抽取实体与关系，写入图数据库
- 增量写入约束：Graphiti 默认不会自动删除旧节点或冲突事实，写入会持续增长

## 检索策略

- 事件问题 → Episodic 检索
- 偏好/关系/长期事实 → Graph 检索
- 模糊问题 → 先 Episodic，再 Graph 补全
- 使用检索路由：先判问题类型，再决定检索源

## 更新与遗忘

- 事件记忆衰减：时间越久权重越低
- 事实可覆盖：新信息标记替代旧偏好
- 支持反向记忆：用否定记忆纠错
- Graph 治理策略：对长期未命中的节点做降权或归档，必要时清理

## 架构关键组件

- 事件写入器：将行为与对话事件写入 MongoDB
- 记忆输入整理器：筛选关键事件文本与去噪，供 Graphiti 抽取入图
- Graphiti 抽取模块：自动从文本抽取实体与关系，写入图数据库
- 记忆冲突处理：新事实可覆盖旧事实，保留时间线
- 图规模治理器：定期评估节点规模、命中率与衰减策略
- 检索路由器：根据问题类型选择 Episodic 或 Graph
