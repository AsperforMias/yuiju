# 记忆模块迭代节奏

本文档描述记忆模块在当前项目中的推荐迭代顺序，目标是：

- 先建立稳定的数据真相层，避免后续反复返工。
- 先让 LLM 能稳定使用记忆，再逐步提升记忆质量。
- 将高复杂度能力拆到后期，控制实现风险。

## 1. 迭代原则

记忆模块的建设顺序不应按“功能看起来重要”排序，而应按“依赖关系”和“返工成本”排序。

推荐遵循以下原则：

- 先统一写入，再升级查询。
- 先建设事件真相层，再建设长期语义层。
- 先让主链路可用，再引入治理和高阶抽取。
- 先建设记忆模块，再建设日记模块。
- 计划模块要尽早纳入，但不要一开始就做成复杂调度系统。

## 2. 总体节奏概览

建议分为五个阶段推进：

1. 阶段一：统一 Episode 写入，打稳事件真相层。
2. 阶段二：建设查询链路，并行验证 Graphiti 最小接入。
3. 阶段三：建设计划模块，补齐当前计划状态与计划历史。
4. 阶段四：将 Graphiti 从 PoC 升级为正式语义层。
5. 阶段五：建设日记模块，形成叙事归档层。

这个顺序的核心原因是：

- 没有统一 Episode，后面的查询、Graphiti、日记都会建立在不稳定输入上。
- 没有稳定查询链路，记忆即使写入了也无法真正被 LLM 用起来。
- 当前项目还没有线上数据包袱，适合尽早验证 Graphiti 是否真正适合本项目。
- 计划模块处于决策链核心，应早于图谱提炼落地。
- 日记依赖 Episode 和计划历史，适合作为后置能力。

## 3. 阶段一：统一 Episode 写入

### 3.1 阶段目标

- 统一行为、对话、计划变更的事件模型。
- 明确 `MemoryEpisode` 是统一事件真相源。
- 为后续检索、Graphiti 提炼、日记生成提供稳定输入。

### 3.2 主要建设内容

- 设计统一的 `MemoryEpisode` 数据结构。
- 在 `world` 中抽出行为 episode 构建器。
- 在 `message` 中抽出对话 episode 构建器。
- 将计划变更写入 `Plan Episode`。
- 明确 `IBehaviorRecord` 为迁移中的兼容层。

### 3.3 为什么第一步先做这个

- 这是所有后续能力的输入基础。
- 这是返工成本最高的一层，越晚统一，后面迁移越痛。
- 这一步完成后，即使后续 Graphiti 和查询还没做好，数据也已经按正确方式沉淀了。

### 3.4 阶段完成标志

- 行为、对话、计划变化均可写入统一 Episode。
- `tick` 与聊天主流程中不再散落拼装记忆结构。
- 后续模块都可以只依赖 Episode，而不依赖旧行为日志结构。

## 4. 阶段二：建设查询链路，并行验证 Graphiti 最小接入

### 4.1 阶段目标

- 让 LLM 可以通过 function tool 按需检索记忆。
- 建立记忆系统的真正使用入口。
- 在没有线上数据包袱的阶段，尽早验证 Graphiti 的适配性。

### 4.2 主要建设内容

- 升级 `memory-search` tool 的输入输出结构。
- 建立统一的 query router。
- 支持 `episode / fact / auto / plan` 等查询意图。
- 统一查询结果格式，避免直接暴露底层原始数据。
- 建设最小可用的 Graphiti PoC。
- 选取少量高价值场景验证 fact 提炼，例如偏好、关系、长期目标倾向。

### 4.3 为什么第二步做查询

- 记忆系统的价值不在于“存进去了”，而在于“模型能稳定查出来并使用”。
- 查询链路越早落地，越容易在真实使用中发现写入结构是否合理。
- 当前项目尚无线上数据迁移成本，适合提前小规模验证 Graphiti。
- 这一阶段的 Graphiti 目标不是完整治理，而是尽快确认“是否值得做、适合做什么”。

### 4.4 阶段完成标志

- LLM 可以通过 `memory-search` 稳定查询行为、对话、计划相关记忆。
- 查询结果已经具备结构化语义，而不是纯搜索结果透传。
- 世界决策链和聊天链路都具备按需查记忆的入口。
- 已有最小 Graphiti 链路可用于验证偏好、关系或长期倾向检索效果。

## 5. 阶段三：建设计划模块

### 5.1 阶段目标

- 将计划从简单字段升级为独立模块。
- 让计划既有当前状态，也有历史轨迹。

### 5.2 主要建设内容

- 建设 Redis 中的 `Plan State`。
- 明确主计划与活跃计划集合。
- 建设 `Plan Manager`，接管计划 proposal 审议与状态流转。
- 将行为 episode 与 `relatedPlanId` 关联起来。
- 对 Redis 中的活跃计划建立清理与收缩机制。

### 5.3 为什么在 Graphiti 之前做计划

- 计划直接影响 tick 决策质量。
- 当前计划状态是即时运行能力，不应被长期语义层阻塞。
- 如果计划模型不稳定，后续从计划历史提炼长期目标倾向也会失真。

### 5.4 阶段完成标志

- 当前计划不再只是 `characterState` 上的简单字段。
- LLM 返回的是计划 proposal，而不是直接覆盖状态。
- 计划历史与行为历史已经可以通过 `planId` 串联。

## 6. 阶段四：将 Graphiti 从 PoC 升级为正式语义层

### 6.1 阶段目标

- 将 Graphiti 从早期验证能力升级为正式的长期语义记忆层。
- 让系统能稳定回答偏好、关系、长期目标倾向这类问题。

### 6.2 主要建设内容

- 将 PoC 阶段的提炼规则正式化。
- 建设 `memory extractor`。
- 建设 `graph sync`。
- 从行为、对话、计划历史中提炼 fact candidate。
- 将 Graphiti 作为长期事实层正式接入查询路由。
- 建立基础治理策略，避免图谱随着试验式写入持续失控。

### 6.3 为什么放到这个阶段

- Graphiti 的适配性可以早验证，但正式化建设仍依赖稳定的 Episode、查询链路和计划模型。
- 计划模块稳定后，再提炼长期目标倾向会更可信。
- 这个阶段再补治理层，能避免 PoC 阶段过早过度设计。

### 6.4 阶段完成标志

- `memory-search` 已能对长期事实进行稳定查询。
- 偏好、关系、长期目标倾向不再完全依赖原始 Episode 回答。
- 事实提炼是异步执行，不阻塞主链路。
- Graphiti 已从“验证性接入”升级为“正式语义层”。

## 7. 阶段五：建设日记模块

### 7.1 阶段目标

- 在记忆系统稳定后，建设独立的日记归档层。
- 既服务玩家查看，也支持悠酱自我回顾。

### 7.2 主要建设内容

- 基于 Episode、Plan Episode、对话 Episode 构建日记输入。
- 建立独立的日记模块与日记存储。
- 提供独立的 `diary-search` tool。

### 7.3 为什么最后做日记

- 日记依赖记忆系统提供高质量、可回溯的输入。
- 日记不直接决定当前行为决策，优先级低于记忆主链路。
- 如果前面的 Episode 和计划历史不稳定，日记内容质量也会很差。

### 7.4 阶段完成标志

- 日记模块与记忆模块边界清晰。
- 悠酱可以查询自己的日记，但日记不混入记忆查询结果。
- 日记成为记忆系统的下游产物，而不是替代记忆系统。

## 8. 不建议的顺序

以下顺序不推荐：

- 一上来先做 Graphiti
  原因：虽然早期适合做最小 PoC，但不适合在写入层和查询层未稳定前直接重投入正式建设。

- 一上来先做日记
  原因：日记质量完全依赖底层记忆数据质量。

- 在查询链路未成型前大量投入记忆写入优化
  原因：很难知道哪些字段真的被 LLM 用到了。

- 让计划模块长期停留在“LLM 每 tick 直接覆盖字段”
  原因：计划会抖动，无法形成稳定目标感。

## 9. 推荐里程碑

为了让节奏更清晰，建议按以下里程碑推进：

1. `M1`：Episode 写入统一
   行为、对话、计划变更都有统一事件模型。

2. `M2`：记忆查询可用
   `memory-search` 可供 LLM 稳定按需调用。

3. `M3`：计划模块落地
   `Plan State + Plan Manager + Plan Episode` 成型。

4. `M4`：Graphiti 正式化
   Graphiti 从最小 PoC 升级为正式长期事实层，长期偏好/关系/目标倾向可稳定检索。

5. `M5`：日记模块上线
   日记与记忆系统形成完整上下游关系。

## 10. 最终结论

推荐的建设顺序是：

`Episode -> Query + Graphiti PoC -> Plan -> Graphiti Formalize -> Diary`

这是当前项目下最稳妥的顺序，因为它优先保证：

- 数据先被正确沉淀
- LLM 先能正确使用
- Graphiti 先被验证，而不是盲目后置或盲目重投入
- 决策核心状态先被建模
- 高阶语义能力后置增强
- 叙事能力作为最终下游产物补齐
