# 记忆系统技术方案

本文档描述记忆系统如何在当前项目中落地，重点关注以下问题：

- 记忆能力放到哪些子包中建设。
- 行为与对话记忆如何写入。
- LLM 如何通过 function tool 查询记忆。
- MongoDB 与 Graphiti 在项目中分别承担什么职责。

## 1. 技术目标

- 在不破坏现有 `world` 与 `message` 主流程的前提下接入记忆系统。
- 基于 MongoDB 建立可回溯的事件记忆。
- 基于 Graphiti 建立可供长期问答与推理的事实记忆。
- 为 LLM 提供稳定、结构化、可扩展的记忆查询工具。
- 保持模块边界清晰，避免 `utils` 子包演变为业务杂糅层。

## 2. 总体落地思路

结合当前代码，记忆系统分为两条主链路：

1. 写入链路  
   `world tick / message chat -> episode writer -> MongoDB -> fact extractor -> Graphiti`

2. 查询链路  
   `LLM function tool -> memory query router -> MongoDB / Graphiti -> structured result -> LLM`

这里的设计原则是：

- `MongoDB` 负责保存“发生过什么”。
- `Graphiti` 负责沉淀“这说明了什么”。
- `LLM` 不直接感知底层存储，而是通过统一工具查询记忆。

## 3. 子包职责划分

记忆系统不作为一个完整业务模块落在 `@yuiju/utils` 中，而是拆为“公共底座 + 业务接入”。

### 3.1 `@yuiju/utils`

负责通用、无业务状态、可跨包复用的记忆基础设施。

建议放置内容：

- memory client 接口与工厂方法。
- 记忆查询 tool 定义。
- episode / search result / fact candidate 等通用类型。
- 记忆查询路由接口定义。
- 通用 schema 校验与参数解析。

### 3.2 `@yuiju/world`

负责世界引擎侧的记忆写入与使用。

建议放置内容：

- 行为 episode 构建与写入适配器。
- tick 决策阶段的记忆查询接入。
- 与角色状态、世界状态相关的记忆上下文组装。

### 3.3 `@yuiju/message`

负责消息服务侧的记忆写入与使用。

建议放置内容：

- 对话窗口 episode 构建与写入适配器。
- 聊天回复阶段的记忆查询接入。
- 用户纠正、偏好表达、关系变化等对话特征的记忆提炼入口。

## 4. 与现有代码的对应关系

当前已有两个明确记忆写入入口：

### 4.1 行为记忆入口

文件：[tick.ts](/Users/yixiaojiu/Code/everything/yuiju/packages/world/src/engine/tick.ts)

现状：

问题：

- 写入逻辑分散在 tick 流程内部，不利于扩展。
- 长期计划变更等关键状态变化只写日志，没有进入记忆系统。
- 当前 `IBehaviorRecord` 更像“行为日志表”，还不足以承担统一记忆事件模型。

### 4.2 对话记忆入口

文件：[chat-session-manager.ts](/Users/yixiaojiu/Code/everything/yuiju/packages/message/src/chat-session-manager.ts)

现状：

- 聊天消息先按时间窗口聚合。
- 窗口结束后调用 `memoryClient.writeEpisode(...)` 写入一条“对话” episode。

问题：

- 对话 episode 主要是消息窗口归档，缺少统一摘要字段。
- 还没有为后续事实提炼提供稳定结构。
- 没有显式区分“原始消息列表”和“可供检索/抽取的摘要文本”。

## 5. 写入链路设计

### 5.1 Episode 作为统一写入单元

记忆系统的第一落点统一为 Episode，所有行为与对话都先写为 Episode，再考虑是否提炼为长期事实。

这里的关键约束是：

- `MemoryEpisode` 应作为统一事件真相源。
- 不再新增一个与 `IBehaviorRecord` 平行、功能重叠的“行为记忆集合”。
- 当前 `IBehaviorRecord` 视为迁移中的兼容层，而不是未来的主记忆模型。

建议 Episode 统一结构如下：

```ts
interface MemoryEpisode {
  id: string;
  source: "world_tick" | "chat";
  type: "behavior" | "conversation" | "plan_update" | "system";
  subjectId: string;
  counterpartyId?: string;
  happenedAt: Date;
  summaryText: string;
  importance: number;
  payload: Record<string, unknown>;
  extractionStatus: "pending" | "processing" | "done" | "skipped" | "failed";
  extractedFactIds?: string[];
  createdAt: Date;
  updatedAt: Date;
}
```

字段说明：

- `payload`：保留结构化原始信息，便于回溯。
- `summaryText`：为检索与事实提炼准备的摘要文本。
- `extractionStatus`：支撑异步提炼与重放。
- `extractedFactIds`：建立 Episode 与图事实的追踪关系。

说明：

- `plan_update` 保留在 Episode 中，但它只负责记录“计划发生过什么变化”。
- 当前生效中的短期计划与长期计划，不应只靠 Episode 回放获得，而应有独立的计划状态模型承载。

### 5.1.1 与 `IBehaviorRecord` 的关系

当前项目已经存在 `IBehaviorRecord`，用于保存行为记录并支持：

- tick 决策前读取最近行为。
- Web 活动流展示。
- 行为历史快速查询。

但它的职责更接近“行为日志读写模型”，而不是统一记忆事件模型，原因包括：

- 只覆盖行为，不覆盖对话、计划变更、系统事件。
- 字段过少，缺少 `payload`、`summaryText`、`importance`、`extractionStatus` 等记忆系统所需信息。
- 无法稳定支撑后续的事实提炼与图谱追踪。

因此技术方案建议：

1. 长期目标：以 `MemoryEpisode` 统一承接所有事件记忆。
2. 迁移期间：保留 `IBehaviorRecord` 作为兼容层或读模型。
3. 后续逐步将 `getRecentBehaviorRecords`、活动流等能力迁移为从 `MemoryEpisode(type=behavior)` 派生查询。

换句话说，`IBehaviorRecord` 不与 `MemoryEpisode` 并列长期存在，而是逐步被统一事件模型吸收。

### 5.2 行为记忆写入

在 `world` 中新增行为 episode 构建器，例如：

- `buildBehaviorEpisode(context, selectedAction, executionResult, durationMin)`
- `buildPlanUpdateEpisode(...)`

写入时机：

- action 执行成功后写行为 episode。
- 长期计划或短期计划有变更时，额外写计划变更 episode。

迁移策略建议：

- 第一阶段允许同时保留现有 `saveBehaviorRecord(...)`，以兼容当前最近行为查询逻辑。
- 新记忆链路以 `MemoryEpisode(type=behavior)` 为主。
- 待查询侧迁移完成后，再考虑将 `IBehaviorRecord` 收敛为兼容查询或派生数据。

行为 episode 的 `payload` 建议包含：

- `tickId`
- `action`
- `reason`
- `executionResult`
- `durationMinutes`
- `location`
- `relatedPlanId`
- `characterStateSnapshot`（仅保留必要摘要）

### 5.3 对话记忆写入

在 `message` 中新增对话 episode 构建器，例如：

- `buildConversationEpisode(counterpartyName, state)`

对话 episode 的 `payload` 建议包含：

- `subjectName`
- `counterpartyName`
- `windowStart`
- `windowEnd`
- `messages`

额外补充字段：

- `summaryText`：对整个对话窗口做摘要，供检索与提炼。
- `importance`：根据对话内容中的偏好、纠正、关系变化等信号动态评估。

### 5.4 异步提炼而非同步入图

不建议在 `tick` 或聊天主流程中同步调用 Graphiti。

建议流程：

1. 业务侧只负责写 Episode。
2. Episode 初始状态为 `pending`。
3. 独立的 `memory extractor` 扫描待处理 Episode。
4. 对有长期价值的 Episode 提炼 fact candidate。
5. 由 `memory graph sync` 将 candidate 写入 Graphiti。

这样做的原因：

- 不阻塞 world tick。
- 不阻塞 message 主链路。
- 抽取规则可迭代、可回放、可重试。

## 6. 计划模块设计

短期计划与长期计划属于记忆系统的重要组成部分，但它们不应只作为简单字符串字段存在。

技术方案建议将计划拆为两层：

1. `Plan State`
   负责表示当前正在生效的计划，是决策时直接消费的运行态数据。
2. `Plan Episode`
   负责记录计划的创建、更新、完成、放弃、替代等历史变化，是计划的可回溯时间线。

### 6.1 为什么单独建设计划模块

当前实现中，计划主要通过状态写入方式直接更新：

- `setLongTermPlan(...)`
- `setShortTermPlan(...)`

这种方式虽然简单，但问题明显：

- 缺少计划生命周期管理。
- 缺少计划与行为之间的显式关联。
- 难以回答“某个行为是在推进哪个计划”。
- 难以判断旧计划是完成、失效还是被新计划替代。

因此，计划不能只被视作普通状态字段，也不能只记录成一条 Episode，而应同时具备“当前状态 + 历史轨迹”。

### 6.2 Plan State 模型

建议为短期计划与长期计划建立统一的活动计划结构，例如：

```ts
interface ActivePlan {
  id: string;
  type: "short_term" | "long_term";
  goal: string;
  status: "active" | "completed" | "abandoned" | "superseded";
  createdAt: Date;
  updatedAt: Date;
  expiresAt?: Date;
  parentPlanId?: string;
  reason?: string;
}
```

建议职责：

- `long_term plan`：表达较长时间跨度的持续目标。
- `short_term plan`：表达当前阶段的具体执行安排。
- 短期计划可以通过 `parentPlanId` 关联到长期计划。

### 6.2.1 Plan State 存储方案

`Plan State` 属于运行时高频读取的当前状态，建议继续基于 Redis 存储，而不是放入 MongoDB。

原因：

- tick 决策时需要低延迟读取当前计划。
- 计划当前态本质上属于运行时状态，而不是历史归档。
- Redis 已经是当前角色状态的核心存储介质，接入成本最低。

建议存储策略：

1. `character_state` 只保留当前主计划与活跃计划集合引用
2. 活跃计划完整内容存入独立的 Redis `plan_state` 结构
3. 计划历史不长期保存在 Redis，而是统一通过 MongoDB 中的 `Plan Episode` 承接

建议示例：

```ts
interface CharacterPlanState {
  primaryLongTermPlanId?: string;
  primaryShortTermPlanId?: string;
  activeLongTermPlanIds: string[];
  activeShortTermPlanIds: string[];
}

interface ActivePlan {
  id: string;
  type: "short_term" | "long_term";
  goal: string;
  steps?: string[];
  status: "active" | "completed" | "abandoned" | "superseded";
  createdAt: string;
  updatedAt: string;
  expiresAt?: string;
  parentPlanId?: string;
  source: "llm" | "system" | "user";
  reason?: string;
}
```

Redis 落地建议：

- `character_state.primaryLongTermPlanId`
- `character_state.primaryShortTermPlanId`
- `character_state.activeLongTermPlanIds`
- `character_state.activeShortTermPlanIds`
- `active_plan:{planId}` -> `ActivePlan`

这样做的好处：

- 允许多个计划并存，而不是被迫收敛为单个计划。
- 同时保留“主计划”概念，避免 tick 决策时上下文过于发散。
- 活跃计划结构可以独立扩展，不影响其他角色状态。
- 行为 episode、计划 episode、记忆查询都可以稳定引用 `planId`。

### 6.2.2 Redis 膨胀控制与计划清理

Redis 中的计划数据不应作为永久存储，而应被视为“当前运行态缓存”。

建议规则：

- `active` 状态计划保留在 Redis 中。
- `completed / abandoned / superseded` 状态计划从活跃计划集合移除。
- 已结束计划的 `active_plan:{planId}` 可设置短期 TTL，或在状态迁移后直接删除。
- 所有长期历史都通过 `Plan Episode` 落入 MongoDB，避免 Redis 持续膨胀。

建议清理策略：

1. 计划进入终态时：
   - 从 `activeLongTermPlanIds` 或 `activeShortTermPlanIds` 中移除
   - 如它是主计划，则同步清空或切换新的主计划引用
2. 对已结束计划：
   - 短期保留 Redis 缓存用于最近查询和链路联动
   - 超过 TTL 后删除 Redis 明细
3. 历史查询统一走 MongoDB `Plan Episode`

因此，Redis 中的 `active_plan:{planId}` 只是活跃计划缓存，不承担永久历史保存职责。

### 6.3 Plan Episode 模型

计划历史继续写入 `MemoryEpisode`，但建议细分计划事件类型，例如：

- `plan_created`
- `plan_updated`
- `plan_completed`
- `plan_abandoned`
- `plan_superseded`

建议 `payload` 至少包含：

- `planId`
- `planType`
- `goal`
- `statusBefore`
- `statusAfter`
- `reason`
- `parentPlanId`

这样可以支持后续回溯：

- 某个长期计划是什么时候建立的。
- 为什么这个计划被改写。
- 某个短期计划是否已经完成。

### 6.4 Plan State 与 Episode 的关系

边界建议如下：

- `Plan State`：当前生效值，供 tick 决策直接使用。
- `Plan Episode`：变更历史，供回溯与检索使用。
- `Graphiti`：沉淀稳定的长期目标倾向，而不是承载当前计划状态。

也就是说：

- “现在悠酱的短期计划是什么” -> 查 `Plan State`
- “她上周为什么修改了计划” -> 查 `Plan Episode`
- “她长期倾向于准备考试/保持规律生活” -> 查 `Graphiti`

### 6.5 与 tick 的集成方式

在 [tick.ts](/Users/yixiaojiu/Code/everything/yuiju/packages/world/src/engine/tick.ts) 中，计划相关逻辑建议调整为：

1. 当 LLM 返回新的短期/长期计划建议时，先交给 `Plan Manager` 处理，而不是直接覆盖状态。
2. `Plan Manager` 决定是创建、更新、替代还是忽略计划。
3. 若计划状态实际发生变化，则更新 `Plan State` 并写入对应的 `Plan Episode`。
4. 行为执行时，将当前命中的 `planId` 写入 behavior episode 的 `relatedPlanId`。

这样可以把“计划”和“行为”在时间线上连起来，后续能够回答：

- 最近这些行为是在推进哪个计划。
- 哪个计划产生了最多执行行为。
- 哪个计划最终没有被执行下去。

### 6.5.1 当前方案与演进方向

当前实现是：

- LLM 在行为决策时顺带返回 `updateLongTermPlan` / `updateShortTermPlan`
- 系统直接调用 `setLongTermPlan(...)` / `setShortTermPlan(...)`

这种实现简单，适合作为起点，但不适合作为长期方案，主要问题有：

- 行为决策和计划更新被绑定在同一次输出中，任务耦合过重。
- 计划容易随着每个 tick 的局部上下文频繁波动。
- 系统缺少计划更新门槛，无法区分“补充旧计划”“替代旧计划”“完成后进入下一阶段”。
- 计划变化会产生大量低质量 `plan_update` 事件，影响后续记忆治理。

因此建议演进为：

1. LLM 在 tick 中可以继续返回计划相关信息，但语义改为 `plan proposal`。
2. `Plan Manager` 负责审议该 proposal，而不是直接落地。
3. 只有当 proposal 通过规则判断后，才更新 `Plan State` 并写入 `Plan Episode`。

### 6.5.2 Plan Manager 职责

建议新增 `Plan Manager`，负责计划生命周期管理与计划提案落地。

职责包括：

- 判断 LLM proposal 是创建、更新、替代还是忽略。
- 维护计划状态流转：`active -> completed / abandoned / superseded`
- 为新计划生成稳定 `planId`
- 将旧计划与新计划建立关联
- 控制计划更新频率，避免高频抖动
- 输出计划变更结果供 tick 与记忆模块使用

建议 `Plan Manager` 的输入输出语义：

- 输入：当前 `Plan State` + LLM `plan proposal` + 角色当前状态
- 输出：计划变更决议 + 新的 `Plan State` + 需要写入的 `Plan Episode`

### 6.5.3 计划生成机制的阶段演进

建议按三个阶段建设：

1. 阶段一：保留当前模式
   LLM 在行为决策时顺带返回计划更新建议，先满足基础能力。

2. 阶段二：引入 `Plan Manager`
   LLM 输出从“直接写计划”改为“提出计划提案”，系统决定是否采纳。

3. 阶段三：拆分计划更新触发机制
   计划更新不再每个 tick 都参与，而是在特定条件下触发，例如：
   - 当前计划完成
   - 当前计划连续受阻或失败
   - 角色状态出现明显变化
   - 用户显式干预

长期目标应是第 3 阶段，即将“行为选择”和“计划审议”分成两类不同频率的认知任务。

### 6.6 与记忆查询的关系

计划也属于记忆查询的一部分，但其查询路径需分层处理：

- 当前计划查询：优先读 `Plan State`
- 历史计划查询：查 `MemoryEpisode(type=plan_update)` 或细分计划事件
- 长期目标倾向查询：查 Graphiti

后续 `memory-search` tool 可考虑扩展：

- `memoryType = "plan"`：计划专用查询
- `memoryType = "auto"`：由路由器自动判断是否需要查计划状态

### 6.7 目录结构建议

计划模块建议按职责拆分：

- `packages/world/src/plan/`
  - `plan-state.ts`
  - `plan-manager.ts`
  - `plan-types.ts`
- `packages/world/src/memory/`
  - `build-plan-episode.ts`

其中：

- `plan-state.ts` 负责当前计划状态管理。
- `plan-manager.ts` 负责计划更新、状态流转、过期判断。
- `build-plan-episode.ts` 负责把计划变化写入记忆系统。

## 7. 查询链路设计

### 7.1 查询入口定位

记忆不是通过预先塞满 prompt 来使用，而是通过 LLM 按需调用 function tool。

这意味着：

- LLM 决定是否需要查询记忆。
- tool 决定查询哪类记忆。
- 存储层只提供原始检索能力，不直接暴露给业务 prompt。

### 7.2 现有 `memory-search` tool 的定位

文件：[memory-search.ts](/Users/yixiaojiu/Code/everything/yuiju/packages/utils/src/llm/tools/memory-search.ts)

现状：

- 只有 `query` 一个参数。
- 直接调用 `client.searchMemory(...)`。
- 返回值是底层 memory service 的原始结果。

问题：

- 缺少检索意图表达。
- 无法区分事件查询与事实查询。
- 缺少时间范围、对象约束等结构化过滤能力。
- 返回结果对 LLM 不够友好。

### 7.3 记忆查询工具升级方案

建议将 `memory-search` 升级为统一记忆检索入口，而不是单纯文本搜索框。

建议输入结构：

```ts
{
  query: string;
  memoryType?: "auto" | "episode" | "fact";
  timeRange?: "today" | "recent_3_days" | "recent_week" | "all";
  counterpartyName?: string;
  topK?: number;
}
```

字段含义：

- `query`：用户问题或需要检索的记忆描述。
- `memoryType`：
  - `episode`：显式查事件记忆。
  - `fact`：显式查长期事实。
  - `auto`：由查询路由器自动判断。
- `timeRange`：限制时间窗口，适合行为回溯类问题。
- `counterpartyName`：限制某个对象或关系主体。
- `topK`：召回上限。

### 7.4 查询路由规则

在 `@yuiju/utils` 中定义统一查询路由接口，例如：

```ts
interface MemoryQueryRouter {
  search(input: MemorySearchInput): Promise<MemorySearchResult[]>;
}
```

路由规则建议如下：

- `episode`：查 MongoDB 中的 Episode。
- `fact`：查 Graphiti 中的长期事实。
- `auto`：通过关键词和约束自动选择。

典型例子：

- “今天早上做了什么” -> `episode`
- “喜欢喝什么” -> `fact`
- “最近为什么老去咖啡馆” -> `episode + fact`

### 7.5 查询结果统一结构

不建议把底层搜索结果原样返回给 LLM。

建议统一为：

```ts
interface MemorySearchResult {
  source: "episode" | "fact";
  score: number;
  summary: string;
  happenedAt?: string;
  validFrom?: string;
  validTo?: string;
  evidenceIds: string[];
  metadata?: Record<string, unknown>;
}
```

这样可以让 LLM 明确知道：

- 这条结果来自时间线还是长期事实。
- 它是某次事件，还是稳定偏好/关系。
- 背后有哪些证据可以引用。

### 7.6 日记查询的边界

未来计划建设独立的日记模块，但不建议将日记查询直接并入 `memory-search`。

原因：

- 记忆查询面向“发生了什么、当前知道什么、当前倾向是什么”。
- 日记查询面向“过去如何被叙述和回顾”。
- 记忆结果通常是事件片段、事实片段、证据引用。
- 日记结果通常是完整条目、整段叙事或阶段性总结。

因此建议：

- `memory-search`：继续负责事件记忆、事实记忆、计划记忆查询。
- `diary-search`：作为独立 function tool，负责日记查询。

两者在工程上可以共享基础查询能力，但不应在 LLM 接口层混为一个工具。

## 8. MongoDB 与 Graphiti 的职责边界

### 8.1 MongoDB

MongoDB 存储 `MemoryEpisode`，是项目中的事件记忆真相层。

这里建议最终收敛为单一主事件集合，例如 `memory_episodes`。

现有 `BehaviorRecord` 不再作为长期主记忆模型扩展，而是在迁移期承担兼容职责。

适合回答的问题：

- 今天早上做了什么？
- 上次和谁聊天是什么时候？
- 最近三天都去过哪些地方？

### 8.2 Graphiti

Graphiti 存储由 Episode 提炼出的长期事实，是项目中的语义记忆层。

适合回答的问题：

- 悠酱喜欢什么？
- 悠酱和谁关系亲近？
- 悠酱最近有什么长期倾向？

### 8.3 重要约束

- 原始 Episode 不直接无脑入图。
- Graphiti 只接收有长期价值的事实候选。
- 所有图事实都应能回溯到一个或多个 Episode。

## 10. 分阶段实施建议

### 阶段 1：统一 Episode 写入

- 抽出行为与对话 episode 构建器。
- 统一 Episode 结构。
- 保留现有 memory service，先确保写入链路规范化。
- 将计划变更写入统一的 `Plan Episode`。
- 继续兼容当前“LLM 直接返回计划更新”的简单模式。

### 阶段 2：升级记忆查询工具

- 扩展 `memory-search` 输入输出结构。
- 建立 query router。
- 让 LLM 可以按 `episode/fact/auto` 查询。
- 将“当前计划查询”纳入查询路由设计。
- 引入 `Plan Manager`，让计划更新改为 proposal 审议模式。

### 阶段 3：接入 Graphiti 提炼链路

- 建立 extractor 与 graph sync。
- 从对话与行为 Episode 中提炼长期事实。
- 从计划历史中提炼长期目标倾向。
- 完成 MongoDB -> Graphiti 的异步同步。

### 阶段 4：在业务流程中正式使用

- 在世界决策链中引入按需记忆查询。
- 在聊天回复链中引入按需记忆查询。
- 在 tick 中完成 `Plan State + Plan Episode` 的联动。
- 将计划更新触发从“每个 tick 顺带处理”逐步收敛为“按条件触发”。
- 基于真实使用情况继续优化召回与摘要质量。

### 阶段 5：建设日记模块

- 在记忆系统稳定后，基于 Episode 与计划历史建设独立日记模块。
- 为 LLM 提供独立的 `diary-search` 工具，支持悠酱回顾自己的日记。
- 保持“记忆系统”和“日记系统”在查询接口层的边界清晰。

## 11. 关键结论

- 记忆系统的技术落地应建立在现有 `tick` 与 `chat session` 两个入口之上，不需要重构主流程。
- `@yuiju/utils` 负责公共记忆底座，不负责完整业务实现。
- `MongoDB` 负责统一的 `MemoryEpisode`，`Graphiti` 负责长期事实。
- `IBehaviorRecord` 与 `MemoryEpisode` 不应长期平行共存，前者应逐步收敛为兼容层或派生读模型。
- 计划模块应从“简单字段”升级为“Plan State + Plan Episode + 长期语义沉淀”的三层模型。
- `Plan State` 建议独立存储在 Redis 中，`character_state` 仅保留活动计划引用。
- “LLM 在行为决策时顺带返回新计划”可以保留为起点，但长期必须升级为 `plan proposal + Plan Manager` 的机制。
- 日记应作为独立模块建设，与记忆系统共享底层事件来源，但在 LLM 接口层通过独立的 `diary-search` 工具暴露。
- 记忆查询必须作为方案核心，由 LLM 通过 function tool 按需调用。
- 技术方案的重点不是“把记忆存进去”，而是“让 LLM 稳定地查出来并正确使用”。
