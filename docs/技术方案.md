# Yuiju World 2.0 技术实现方案

## 1. 总体架构

本方案旨在实现“日程表”、“自然时钟”及“世界事件驱动”机制。
采用 **“时间切片 (Time Slicing) + 动态时长 (Dynamic Duration)”** 的方案。

### 1.1 核心思想

- **无异步事件流**：所有事件（日程、闹钟）都是基于时间的静态定义，或者在 `tick` 执行时同步检查。
- **时间切片**：`tick` 函数在计算 Action 的持续时间时，不仅考虑 Action 本身的自然结束时间（如睡满 8 小时），还会检查未来的日程表中是否有更早发生的事件（如闹钟）。如果存在，则将 duration 截断到该事件发生的时间点。
- **被动唤醒**：当 Runner 等待完这个被截断的时长后，会再次调用 `tick`。此时，系统检查当前时间，发现“闹钟响了”或“上课铃响了”，从而生成对应的 Context 供 LLM 决策。

### 2.2 Tick 逻辑改造

`tick` 函数负责决策 Action 并计算时长。

```typescript
// packages/world/src/engine/tick.ts

export async function tick(): Promise<number> {
  const now = worldState.getCurrentTime();

  // 1. 获取当前触发的事件
  const triggeredEvents = scheduleSystem.getTriggeredEvents(now);

  // 2. 构建 Context
  const context = {
    events: triggeredEvents,
    // ...
  };

  // 3. LLM 决策 -> Action
  const action = await decideAction(context);
  // 例如：action = { type: 'SLEEP', idealDuration: 480 }

  // 4. 动态计算时长 (Dynamic Duration)
  // 获取下一个关键时间点 (如 07:30 的闹钟)
  const nextKeyTime = scheduleSystem.getNextKeyTime(now);

  // 计算距离下一个事件还有多久
  const minutesUntilEvent = nextKeyTime ? nextKeyTime.diff(now, 'minute') : Infinity;

  // 取最小值：Action 理想时长 vs 距离下一个事件的时长
  const actualDuration = Math.min(action.idealDuration, minutesUntilEvent);

  // 5. 执行 Action (更新状态等)
  await executeAction(action);

  // 6. 返回实际等待时长
  return actualDuration;
}
```

### 2.3 Runner 逻辑

Runner 保持极简，完全依赖 `tick` 返回的时长进行等待。

```typescript
// packages/world/src/engine/runner.ts

export async function startRealtimeLoop() {
  while (!stopped) {
    worldState.updateTime();

    // tick 内部已经处理了所有逻辑，并返回了安全的休眠时长
    // 这个时长保证了不会错过下一个日程点
    const durationMin = await tick();

    // 如果 durationMin 为 0 (例如事件处理完毕，或者需要立即再次决策)，则不等待
    if (durationMin > 0) {
      await setTimeout(durationMin * 60 * 1000);
    }
  }
}
```

### 2.4 场景推演

**场景：睡眠与闹钟**

1.  **23:00** - AI 决定睡觉 (Sleep)。

    - `idealDuration` = 8 小时 (目标 07:00 醒)。
    - `nextKeyTime` = 07:30 (闹钟)。
    - `minutesUntilEvent` = 8.5 小时。
    - `actualDuration` = min(8h, 8.5h) = 8h。
    - Runner 等待 8 小时。

2.  **07:00** - Runner 醒来 (自然醒)。
    - `tick` 执行。`getTriggeredEvents` 为空。
    - AI 状态：睡饱了。
    - AI 决定起床或发呆。

**场景：熬夜与闹钟**

1.  **02:00** - AI 决定睡觉 (Sleep)。

    - `idealDuration` = 8 小时 (目标 10:00 醒)。
    - `nextKeyTime` = 07:30 (闹钟)。
    - `minutesUntilEvent` = 5.5 小时。
    - `actualDuration` = min(8h, 5.5h) = 5.5h。
    - Runner 等待 5.5 小时。

2.  **07:30** - Runner 醒来 (被截断)。
    - `tick` 执行。`getTriggeredEvents` 返回 `[ALARM_CLOCK]`.
    - Context 包含“闹钟响了”。
    - LLM 决策：是“起床”还是“关闹钟继续睡”。

### 2.5 优缺点分析

- **优点**：
  - 架构极其简单，无需引入 Redis/BullMQ。
  - 纯同步逻辑，易于测试和调试。
  - 完全符合“日程表”这种预定义时间的业务形态。
- **缺点**：
  - 难以响应**真正的随机突发事件**（如用户突然发了一条消息）。对于这种情况，可能需要 Runner 在等待时支持被外部信号 `interrupt`，或者保持较短的轮询间隔（如每分钟 tick 一次，或者在长等待时使用 `Promise.race([sleep, userMessageSignal])`，但这超出了当前方案的讨论范围，目前聚焦于 Schedule）。

2.  **Tick Refactor**: 引入 `ScheduleSystem`，实现动态时长计算逻辑。
