# 技术方案

## 天气系统

### 设计目标

天气系统作为世界背景状态的一部分，由 `@yuiju/world` 统一负责生成与更新，`@yuiju/message`、`@yuiju/web` 等其他模块只消费当前天气状态，不参与天气计算。

第一版天气系统的设计目标如下：

- 天气具有稳定的时间节奏，不会频繁抖动；
- 天气变化具备一定连续性和合理性，而不是纯随机跳变；
- 其他服务可以直接读取当前天气状态；
- 天气变化可以被记录下来，为后续 diary、memory、LLM 上下文提供依据；
- 当前方案先聚焦天气状态本身，不直接影响 action，也不提供天气预报能力。

### 模块职责

在 `packages/world/src/engine` 中新建 `weather` 模块，用于收敛天气相关逻辑。`weather` 模块本身只对外暴露天气领域能力，不在模块内部启动 cron。该模块负责以下职责：

- 计算当前时间片对应的天气结果；
- 在 `world` 服务启动时校正当前天气状态；
- 将最新天气快照写入 Redis；
- 在天气发生实际变化时写入 `memory/episode`。

项目入口负责统一初始化调度逻辑，例如在 `main.ts` 中完成以下工作：

- 启动时调用一次天气状态校正；
- 注册天气 cron 定时任务；
- 统一管理后续其他定时任务的启动逻辑。

世界时间暂时保持与真实时间一致，不单独通过 cron 维护。天气系统只基于当前真实时间判断所处时间片并生成天气结果。

### 更新机制

天气通过 cron 定时任务按 6 小时时间片更新，每天固定在 `00:00`、`06:00`、`12:00`、`18:00` 执行。由于后续项目中可能还会出现其他定时任务，天气任务建议直接接入统一的 cron 调度方案。cron 包第一版建议使用 `node-cron`，原因是其能力足够覆盖当前固定时间点调度场景，使用简单，并且支持显式时区配置。

除定时触发外，`world` 服务启动时还需要执行一次天气状态校正，避免服务重启后天气状态过期。启动校正规则如下：

- 如果 Redis 中当前天气仍然落在有效时间片内，则继续沿用；
- 如果当前天气已过期，则立即补算当前时间片天气并写回；
- 校正完成后再进入正常 cron 调度。

### 生成策略

天气采用基于算法的更新方式，不使用 LLM 直接决定天气结果。第一版生成逻辑以“季节权重 + 上一时段天气惯性”为核心：

- 季节权重用于控制不同月份更容易出现的天气类型；
- 上一时段天气惯性用于保证相邻时间片之间的变化更自然；
- 生成结果应尽量避免出现过于突兀的跳变；
- `temperatureLevel` 需要与天气类型一起生成，用于描述当前时段的体感温度氛围。

推荐将天气生成理解为一个轻量状态流转过程：先根据月份确定基础天气倾向，再结合上一时间片天气对候选结果进行收敛，最后得到当前时间片天气。

### 状态结构

第一版天气状态至少包含以下字段：

- `type`：天气类型；
- `temperatureLevel`：体感温度等级；
- `periodStartAt`：当前天气时间片开始时间；
- `periodEndAt`：当前天气时间片结束时间；
- `updatedAt`：最近更新时间。

其中：

- `type` 用于表达晴、阴、雨、雪等天气状态；
- `temperatureLevel` 用于表达寒冷、清凉、舒适、温暖等体感温度氛围；
- `periodStartAt` 和 `periodEndAt` 用于标识当前天气所属时间片，便于启动校正与跨服务消费。

### 存储与历史

当前天气快照存储在 Redis 中，供各服务实时读取。Redis 中只保留当前有效天气，不承担完整历史记录职责。

当天气类型或 `temperatureLevel` 发生变化时，额外写入 `memory/episode`，用于保留天气变化历史，并为后续 diary、memory、LLM 上下文提供依据；如果天气没有实际变化，则不写入 episode，避免产生噪音数据。

天气历史第一版不单独设计独立 Mongo 集合，而是直接复用现有 `memory/episode` 体系，降低系统复杂度并保持事件记录入口统一。

### 设计边界

第一版天气系统暂不包含以下能力：

- 天气对 action 的直接影响；
- 天气预报；
- 基于 LLM 的天气生成；
- 虚拟世界时间与真实时间解耦；
- 独立的天气历史统计模型。
